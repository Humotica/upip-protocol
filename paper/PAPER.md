# 1. Introduction

The reproducibility crisis in computational science is well-documented.
Papers describe methods, publish code, and list dependencies — yet
reproduction often fails due to implicit state, environment drift, or
incomplete process definitions.

We argue the problem is architectural: existing approaches treat
reproducibility as documentation when it should be treated as
**verification**. A process is reproducible not because someone wrote
down the steps, but because re-executing those steps produces the same
cryptographic hash.

UPIP formalizes this insight as a five-layer protocol. Each layer
captures one dimension of process integrity, producing an independent
hash. The complete stack hash represents the full integrity chain from
starting state through execution to result.

## 1.1 The Self-Validation Property

This paper is itself a UPIP artifact. The document you are reading was
generated by `produce_paper.py`, a deterministic script. This script was
exported as a UPIP bundle:

```bash
tibet-triage upip-export -o paper.upip.json -s . \
  --title "UPIP Self-Validating Paper" \
  -- python3 paper/produce_paper.py
```

You can reproduce it:

```bash
tibet-triage upip-reproduce paper.upip.json
```

If the reproduction succeeds, the paper is proven correct — not by
peer review, but by cryptographic verification. The paper does not
merely *describe* the protocol; it *is* the protocol in action.

# 2. Protocol Specification

## 2.1 Layer Architecture

UPIP defines five layers, each capturing one dimension of process
integrity:

| Layer | Name | Captures | Hash Format |
|-------|------|----------|-------------|
| L1 | STATE | Starting state (git commit or file manifest) | `git:<sha>` or `files:<sha256>` |
| L2 | DEPS | Python version + exact package versions | `deps:<sha256>` |
| L3 | PROCESS | Command, intent, actor | (definitional) |
| L4 | RESULT | Exit code, stdout, file changes | `sha256:<hash>` |
| L5 | VERIFY | Cross-machine verification record | (verdict) |

The stack hash combines all layers:

```
stack_hash = "upip:" + SHA256(L1.hash + "|" + L2.hash + "|" + L3.cmd + "|" + L4.hash)
```

## 2.2 L1 — STATE: The Starting Point

The critical question UPIP answers first: **"from what state?"**

A process cannot be reproduced if the starting state is undefined.
L1 captures this state through auto-detection:

- **Git repository**: commit SHA, branch, dirty flag
- **File directory**: SHA-256 manifest of every file
- **Container image**: image digest
- **Empty**: no state required (pure computation)

## 2.3 L2 — DEPS: The Environment

Dependencies are captured as the exact output of `pip freeze`, hashed
as a single string. This records not just direct dependencies but the
full transitive closure.

Note: L2 is informational for reproduction. Different machines may have
different Python versions or package sets. What matters is whether L4
(the result) matches — not whether L2 is identical.

## 2.4 L3 — PROCESS: The Intent

L3 records both the mechanical command and the human intent:

- `command`: the exact shell command to execute
- `intent`: why this process exists (human-readable)
- `actor`: who or what initiated the process

Intent is not verified cryptographically — it provides context for
human understanding. The command is what gets executed.

## 2.5 L4 — RESULT: The Outcome

L4 captures everything that changed:

- Exit code (0 = success)
- Standard output and error
- File diff hash (SHA-256 of added/changed/removed files)
- Complete result hash combining all of the above

The result hash is the primary comparison target for reproduction.

## 2.6 L5 — VERIFY: The Verdict

When a bundle is reproduced on another machine, L5 records:

- Which machine reproduced it
- Whether the result hash matches (isomorphic)
- The divergence score (0.0 = identical, 1.0 = total divergence)
- The reproducing machine's environment

A process is **REPRODUCIBLE** if and only if the L4 result hash
from reproduction matches the original L4 result hash.

# 3. Implementation

UPIP is implemented in `tibet-triage`, a Python package available on
PyPI:

```bash
pip install tibet-triage>=0.3.0
```

The implementation provides:

- `tibet-triage upip-export`: Capture a process as a UPIP bundle
- `tibet-triage upip-reproduce`: Reproduce a UPIP bundle on any machine

Both commands execute within an **airlock sandbox** — an isolated
filesystem environment that tracks all file-level side effects through
SHA-256 before/after snapshots.

# 5. The Meta-Property

This paper exhibits a unique property: **it is both the description and
the proof of the system it describes.**

```
┌─────────────────────────────────────────────────┐
│  produce_paper.py generates PAPER.md            │
│  PAPER.md describes UPIP                        │
│  UPIP bundles produce_paper.py as proof          │
│  Reproducing the bundle regenerates PAPER.md    │
│  Which describes how it was generated            │
│  ∴ The paper validates itself                    │
└─────────────────────────────────────────────────┘
```

This circularity is not a logical flaw. It is a **fixed point**: the
paper is a stable artifact of the process it describes. Any modification
to the process changes the paper, and any modification to the paper
breaks the verification hash. The paper and the process are
cryptographically bound.

Traditional papers say: *"Trust us, it works."*
This paper says: *"Don't trust us. Reproduce it."*

# 6. Related Work

- **Docker/containers**: Capture environment but not process intent or
  starting state. A Dockerfile is not a proof — it's a recipe.
- **Zenodo**: Archives artifacts with DOI but does not verify
  reproducibility. Archiving is not verification.
- **ReproZip**: Captures system calls for replay. Excellent for
  environment capture, but does not define a formal integrity protocol.
- **TIBET**: Provides provenance tokens (ERIN/ERAAN/EROMHEEN/ERACHTER).
  UPIP extends TIBET with process-level integrity verification.

UPIP is complementary to all of the above. A UPIP bundle can be
archived on Zenodo, run inside a Docker container, and carry TIBET
provenance tokens.

# 7. Conclusion

UPIP transforms reproducibility from documentation into verification.
The five-layer protocol captures everything needed to reproduce a
computational process: starting state, dependencies, process definition,
result, and cross-machine verification.

The strongest argument for the protocol is this paper itself. It was
not written — it was computed. And you can verify that computation on
your own machine.

```bash
pip install tibet-triage>=0.3.0
tibet-triage upip-reproduce paper.upip.json
```

If the paper appears, the protocol works. Q.E.D.
